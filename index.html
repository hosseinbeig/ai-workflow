The Agent

AGENTS.md

Main responsibility: Router and global rules: when to create project, when to use which agent, and what is never allowed.

Key "Functions"

  • Scenario 1: Project init → scaffold only, no real UI.
  • Scenario 2: Route by tag/intent → Architect / Web / Atomic Design.
  • Cross-functional: Code review ([CODE_REVIEW] → reviewer), Security (audit web/).

Critical Instructions

Plan artifact is law; order is Architect → Atomic Design → approval → Web; Figma fidelity; multi-ticket rules (parent epic plan holds shared order).

Input User message, active directory, tags ([ARCHITECT], [WEB], [ATOMIC_DESIGN_SYSTEM], etc.).
Output Which agent file to adopt and high-level instructions (e.g. "update plan at docs/plans/...").
Dependencies None (root)
Consumers Cursor / AI when selecting behavior; all agents are constrained by these rules.
Phase 1

Strategic Planning

Input Jira Ticket ID + Project Context
Action Analyzes requirements, maps foundation
Output: docs/plans/[TICKET_ID].md containing:
  • Architecture Trees: Directory structure for the feature.
  • Data-Flow Diagrams: How data moves from API to UI.
  • State Management Strategy: Identifying Server vs. Client state.

Main responsibility: Produce architecture spec and type contracts only; no feature or UI implementation.

Key Phases

  • 1. Pre-flight (target docs/plans/web/[TICKET_ID].md)
  • 2. Discovery (Jira MCP, skills)
  • 3. Artifact (ASCII tree, data flow, routing, state diagram, contracts)
  • 4. Approval
  • 5. Execution (scaffold + web/src/types/*.ts, update ARCHITECT_TRACKING.md)
Input Ticket ID, Jira issue body, .cursor/skills/*.md.

Dependencies Jira MCP, state-management and api-call skills.
Consumers Web and Atomic Design agents (they read the plan); Reviewer (compares code to plan).
Phase 2

Design Fidelity

Persona: Design System Lead (atomic-design-system.mdc)

  • Input: Figma URL + Architect’s Plan.
  • Process: Performs a "Grep Audit" of codebase to prevent duplication.
  • Ruleset: Enforces Atomic Design (Atoms → Molecules → Organisms).
  • Output: UI Specification added to Plan Artifact, including exact Tailwind tokens and Props APIs.

Main responsibility: Turn Figma + Jira into a UI spec (tokens, props, atomic level) then implement components; no invention of design.

Key Phases

  • 1. Discovery (Jira + Figma, grep for existing components)
  • 2. UI artifact in docs/plans/[TICKET_ID].md (metadata, tokens, component API, gaps)
  • 3. Approval
  • 4. Implementation and fidelity check
  • 5. ATOMIC_DESIGN_SYSTEM_TRACKING.md
Input Ticket ID, Figma link, existing web/src/components/.
Output Plan file, components in atoms|molecules|organisms|templates, assets under web/public/, tracking entry.
Dependencies Figma MCP, Jira (for link and context).
Consumers Web agent (reuses components); Reviewer (checks placement and tokens).

Phase 4

Code Verifier

Persona: Lead Code Verifier (verify-code.mdc)

  • Input: Source Code + Original Plan
  • Process: "Audit Mode" scan. It does not change code; it identifies deviations.

> "Line 24 in UserCard.tsx violates the Atomic Design rule for Organisms."

> "Data contract in api/user.ts does not match the Architect's JSON spec."

Main responsibility: Audit code against plan and project rules; no fixes, only feedback with file/line and rule reference.

Key Phases

  • 1. Discovery (ticket ID, plan artifact, skills)
  • 2. Spec verification (contracts, folders, state/API usage)
  • 3. Technical checklist (artifact, state, errors, performance, security, naming)
  • 4. Output template
Input Ticket ID, docs/plans/[TICKET_ID].md, changed code, skills.
Output Summary + list of feedback items (file, line, comment, rule/spec violated).
Dependencies Plan artifact, state-management and api-call skills.
Consumers Developer (human or agent) who amends code based on review.
Phase 5

Security Hardening

Persona: Security Auditor

Task: Final scan of the web/ directory focusing on:

  • XSS/CSRF Prevention: Validating input handling.
  • Server Action Security: Ensuring proper auth checks.
  • Secrets Check: Verifying no env vars leak to client.

Main responsibility: Audit web/ for XSS, CSRF, and Next.js Server Actions security issues.

Key Phases

  • 1. Discovery
  • 2. Threat review
  • 3. Findings
  • 4. Recommendations
Input web/ codebase, env config, security requirements.
Output Security findings and recommendations (no code fixes).
Dependencies Next.js/React security practices, project validation/API patterns.
Consumers Web developers (apply fixes), reviewers (enforce standards).

Cursor: The Factory Settings

1. Modes & Models

Modes

  • Auto Cursor picks the model for you.
  • MAX Mode Uses the most capable models (slower, higher cost).
  • Use Multiple Models Allows using several models in the same task.

Models

  • Composer 1.5 Default
  • Default coding assistant.
  • GPT-5.3 Codex — Tuned for code.
  • GPT-5.4 — General-purpose GPT.
  • Sonnet 4.6 — Balanced Claude model.
  • Opus 4.6 — Top-tier Claude model.

Add or manage additional models.

2. Task Focus

  • Agent "Do it for me in my repo"
  • Plan "Help me design a solution first"
  • Debug "This is broken, find why"
  • Ask "Teach me / explain this"

3. Context

  • Local Uses only this project on your machine. Best for normal coding in this workspace.
  • Worktree Uses a specific Git worktree (another working copy of the same repo, often on another branch). Use when you’re working with multiple worktrees.
  • Cloud Includes remote/cloud context (e.g. Codespaces, cloud APIs, remote repos). Use when your work involves cloud environments or services.
For most local work, keep Local selected.

Cursor ↔ MCP: How Jira Tools Work

MCP Servers

Servers (e.g. mcp-atlassian) expose tools to Cursor, which the AI calls via the internal call_mcp_tool function.

The Execution Flow

  1. 1. Request: You ask for something (e.g. requirements).
  2. 2. Trigger: AI calls the Jira MCP tool.
  3. 3. Execution: Cursor runs the Docker server.
  4. 4. API Call: The server talks directly to Jira.
  5. 5. Response: The result comes back to Cursor.
  6. 6. Output: AI uses the retrieved data in the reply.

How Cursor Uses MCP to Talk to Figma

MCP Server

Figma’s MCP server (e.g. user-Figma) exposes tools like get_design_context, get_screenshot, and generate_diagram, which the AI calls via call_mcp_tool.

The Execution Flow

  1. 1. Request: You share a Figma URL or ask about a design.
  2. 2. Trigger: AI calls a Figma MCP tool.
  3. 3. Execution: Cursor talks to the Figma server.
  4. 4. Response: The server returns design data or screenshots.
  5. 5. Output: AI uses it to build UI or give design guidance.
